> Document every major technical decision here with the *why*.

## Frontend: Next.js 15 (App Router)

**Decision:** Next.js 15 with App Router, TypeScript, Tailwind CSS, Framer Motion.

**Why Next.js over plain React (previous portfolio):**
- Server-side rendering (SSR) and Incremental Static Regeneration (ISR) give Google
  fully-rendered HTML — critical for SEO ranking.
- The old React SPA sent an empty `<div>` to crawlers. Next.js sends actual content.
- `generateMetadata()` API makes per-page SEO trivial and CMS-driven.
- Image optimization via `next/image` (WebP, AVIF, lazy loading, blur placeholder).

**Why App Router over Pages Router:**
- Nested layouts reduce code duplication across pages.
- Server Components mean less JavaScript shipped to the browser.
- Better streaming support for perceived performance.

**Why Framer Motion:**
- Declarative animation API that pairs cleanly with React.
- Built-in `useReducedMotion()` hook for accessibility compliance.
- `AnimatePresence` handles page transition animations.

## Backend: Django 5 + Wagtail 6

**Decision:** Django as the base framework, Wagtail as the CMS layer on top.

**Why Django over alternatives (Strapi, Contentful, etc.):**
- Rezwoan already knows Python/Django — no new language to learn for backend maintenance.
- Full control over the database schema and API shape.
- Wagtail is a Django app, not a separate service — one codebase, one deployment unit.

**Why Wagtail over raw Django admin:**
- Wagtail ships a production-quality admin UI designed for content editing, not just data management.
- Built-in `StreamField` for rich content (blog posts with mixed text, images, code blocks).
- Native Headless/API mode (`wagtail.api`) — no extra work to expose REST endpoints.
- Auto-generates `sitemap.xml` and handles image renditions (resize, crop, WebP).
- Image focal point selection in the admin for smart cropping.

**Why not a third-party headless CMS (Contentful, Sanity):**
- Monthly cost for hobby/portfolio use.
- Vendor lock-in — if pricing changes, migrating is painful.
- Less control over API shape and data structure.
- Self-hosting on the Pi keeps everything under one roof.

## API Pattern: Headless (Wagtail API v2 + DRF)

**Decision:** Wagtail's built-in API v2 for page content, custom DRF endpoints for non-page
models (Skills, Testimonials, ContactSubmissions, SiteSettings).

**Endpoint structure:**
```
GET  /api/v2/pages/           → All pages (Wagtail)
GET  /api/v2/pages/{id}/      → Single page
GET  /api/v2/images/          → Media library
GET  /api/skills/             → Skills list (custom DRF)
GET  /api/projects/           → Projects list with filters (custom DRF)
GET  /api/blog/               → Blog posts (custom DRF with pagination)
GET  /api/site-settings/      → Global settings singleton (custom DRF)
GET  /api/testimonials/       → Featured testimonials (custom DRF)
POST /api/contact/            → Contact form submission (custom DRF)
GET  /sitemap.xml             → Auto-generated by Wagtail
GET  /robots.txt              → Served by Django view
GET  /feed/rss/               → RSS feed for blog
```

**CORS:** `django-cors-headers` allows `http://localhost:3000` in dev and `https://rezwoan.me` in prod.

## Database: PostgreSQL

**Decision:** PostgreSQL as the primary database.

**Why not SQLite:**
- SQLite works for development but cannot handle concurrent writes in production.
- Wagtail's image processing and search features perform better on PostgreSQL.
- Rezwoan already uses PostgreSQL in his other projects — consistent tooling.

**Local dev:** Database named `portfolio_db`, user `portfolio_user`.
See `.env.example` for the full connection string format.

## Data Fetching: ISR (Incremental Static Regeneration)

**Decision:** Next.js ISR with `revalidate: 60` on all dynamic pages.

**Why ISR over:**
- **Full SSR on every request:** Too slow, unnecessary for content that changes at most a few times a day.
- **Full static generation:** Rebuilding the entire site every time an admin saves content is overkill.
- **Client-side fetching (useEffect):** Crawlers see empty pages — kills SEO completely.

**ISR means:** Pages are pre-rendered at build time, served from cache, and silently regenerated
in the background when content changes in Wagtail. A visitor always gets a fast cached page; the
*next* visitor after a cache miss gets the fresh version. For a portfolio, 60-second staleness is fine.

## Hosting: Raspberry Pi 5 via Cloudflare Tunnel

**Decision:** Self-host on the existing Pi 5 ("BlackBox") using the existing `rezwoan-pi` Cloudflare Tunnel.

**Why:**
- Zero additional hosting cost.
- Infrastructure already set up and running.
- Cloudflare handles HTTPS, DDoS protection, and global CDN caching at the edge.

**Architecture without Docker (local dev):**
- Backend: `gunicorn` (prod) / `manage.py runserver` (dev)
- Frontend: `next start` (prod) / `next dev` (dev)
- Nginx as reverse proxy in front of both services in production.

**Process management in production (no Docker):**
- Use `systemd` service units for both the Django/Gunicorn process and the Next.js process.
- See `DEPLOYMENT.md` for the exact systemd unit files.

## Settings Split: base / development / production

**Decision:** Three settings files using `python-decouple` for env vars.

```
backend/portfolio/settings/
├── base.py          ← Shared: installed apps, middleware, Wagtail config
├── development.py   ← DEBUG=True, SQLite option, relaxed CORS
└── production.py    ← DEBUG=False, PostgreSQL, strict CORS, WhiteNoise
```

Set which to use with: `DJANGO_SETTINGS_MODULE=portfolio.settings.development`

---

## Frontend Data Layer: Typed API Client

**Decision:** A single `frontend/lib/api.ts` file that is the *only* place that touches the backend URL.
All components import from this file. No raw `fetch` calls anywhere else in the codebase.

**Why:** If the backend URL, API version, or authentication method changes, there is exactly one
file to update. Components stay clean and testable.

## Image Strategy

- **Wagtail admin:** Handles original image storage, generates responsive renditions on request.
- **Next.js `next/image`:** Wraps all images, handles WebP conversion, lazy loading, blur placeholder.
- **OG images:** Generated dynamically using the `@vercel/og` (or `next/og`) API route so social
  shares always look correct even for new blog posts added via CMS.
- **Formats:** Prefer WebP in production. AVIF where supported (Next.js handles this automatically).

## Animation Philosophy

- All entry animations are **scroll-triggered** (Framer Motion `whileInView`).
- Transitions between pages use `AnimatePresence` with a shared layout animation.
- Duration range: **150ms–400ms**. Nothing slower except hero section reveals (max 800ms).
- Every animation checks `useReducedMotion()` and falls back to instant show if true.
- GPU-composited properties only: `transform` and `opacity`. Never animate `width`, `height`,
  `top`, `left` — these cause layout thrash and drop frames.
